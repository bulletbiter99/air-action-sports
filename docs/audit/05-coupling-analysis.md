# 05 — Public/Admin Coupling Analysis

This is the danger zone. Anything listed here is depended on by **both** the public-facing site and the admin code paths. Future phases must not modify these without an explicit, deliberate review.

**Scope clarification.** This file lists assets that cross the public/admin boundary. Things that are heavily depended on by the public side alone (e.g. `src/components/SEO.jsx`, `useEvents`, `siteConfig.js`, `locations.js`) are not coupling assets — they're public-only — and live elsewhere if they need to be tracked. The Explore agent that surveyed the repo for this audit initially conflated those categories; the entries below are the manually-verified subset.

## How "public" and "admin" are defined for this audit

- **Public side (frontend)** = anything routed through React Router's outer `<Layout>` block (everything except `/admin/*`, `/v/*`, `/vendor/*`). Sources in `src/pages/*`. The vendor portal is treated as a third lane and is mentioned where relevant.
- **Public side (backend)** = `worker/routes/*.js` directly (not under `routes/admin/`): `events.js`, `bookings.js`, `webhooks.js`, `waivers.js`, `taxesFees.js`, `feedback.js`, `vendor.js`, `vendorAuth.js`. Plus `worker/index.js` itself (cron, R2 serve, OG rewriter).
- **Admin side** = `src/admin/*` (frontend) and `worker/routes/admin/*` (backend).
