────────────────────────────────────────────────────────────────────
KEY OPERATIONAL POSTURES (M5.5 carry-forward)
────────────────────────────────────────────────────────────────────

- Field rentals are ADMIN-CREATED via /admin/field-rentals/new OR
  AUTO-CREATED via the public /contact form (subject=private-hire/corporate).
  There is no admin-side POST /api/admin/customers endpoint yet;
  phone-intake operators direct-INSERT via SQL or wait for the post-M5.5
  polish batch.

- Recurrence sweep generates child rentals out to a 90-day horizon
  nightly. With 0 active series today, sweep is a no-op.

- COI + lead-stale sweeps fire only when rentals exist with the
  qualifying conditions. With 0 rentals today, sweeps are no-ops.

- Public /contact form: rate-limited via RL_FEEDBACK, honeypot field
  'website' silent-drops bots, subject routes to either field-rental
  lead OR general inquiry. Operator notification email always sent
  (best-effort: failure doesn't block public submission).

- AES-GCM encryption: customers.business_tax_id + .business_billing_address
  columns exist (migration 0046) but the decrypt+render+edit surface
  is DEFERRED to a post-M5.5 polish batch. Stub messages on the
  customer detail page point to "lands in M5.5 B10" — that text is
  inaccurate post-deploy; track for the polish batch.
